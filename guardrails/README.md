# Operator Guardrails for OpenShift

Operator Guardrails are provided as **RHACM PolicySets generated using PolicyGenerator**.

The project provides governance policies that control which OLM operators can be installed on OpenShift managed clusters. The enforcement mechanism uses Kubernetes **Validating Admission Policies (VAP)** with CEL expressions, while the VAP resources themselves are packaged and generated through **RHACM PolicyGenerator**.

Two independent enforcement approaches are available:

* **Blocklist** — deny operators listed in a ConfigMap and allow all others.
* **Allowlist** — allow only approved operators listed in a ConfigMap and deny all others.

Each approach is delivered as a separate RHACM policy and can be applied independently based on your governance and security requirements.

| Delivery Method                            | Directory                                                                   | Requirements                                   |
| ------------------------------------------ | --------------------------------------------------------------------------- | ---------------------------------------------- |
| RHACM PolicyGenerator (VAP-based policies) | `policies/vap/blocklist/`, `policies/vap/allowlist/`, `policies/vap/audit/` | RHACM 2.15+, Kubernetes 1.30+, OpenShift 4.17+ |

## Architecture

```
PolicyGenerator
        |
        v
RHACM Policy / PolicySet
        |
        v
Managed OpenShift Cluster
        |
        v
ValidatingAdmissionPolicy (VAP)
        |
        v
OLM Subscription Validation
```

The RHACM PolicyGenerator creates RHACM `Policy` resources containing the required:

* `ValidatingAdmissionPolicy`
* `ValidatingAdmissionPolicyBinding`
* Configuration `ConfigMap` resources

The Kubernetes API server on the managed cluster performs the actual enforcement.

---

## How It Works

The guardrails intercept OLM `Subscription` resources during admission before an operator installation is allowed.

### Blocklist Flow

```
User creates Subscription
        |
        v
VAP reads blocked-operators ConfigMap
        |
        v
Is spec.name in blocked package list?
       / \
     yes   no
      |     |
    DENY  ALLOW
```

### Allowlist Flow

```
User creates Subscription
        |
        v
VAP reads allowed-operators ConfigMap
        |
        v
Is spec.name in allowed package list?
       / \
     yes   no
      |     |
   ALLOW  DENY
```

## Deployment Model

These guardrails are **not intended to be applied directly using `oc apply`**. The recommended deployment workflow is through RHACM governance:

```
PolicyGenerator
        |
        v
Generate RHACM Policies
        |
        v
Attach Policies to PolicySet
        |
        v
Assign PolicySet to managed clusters
        |
        v
RHACM deploys VAP resources
        |
        v
Kubernetes enforces operator guardrails
```

Available policy implementations:

* `blocklist` — denies installation of explicitly prohibited operators.
* `allowlist` — permits installation only from an approved list.
* `audit` — records modifications to guardrail resources without blocking them.

---

## Managing Operator Lists

Operator approvals and denials are managed through the ConfigMaps packaged within the generated RHACM policies.

### Blocklist

Update:

```
policies/vap/blocklist/blocked-operators-configmap.yaml
```

### Allowlist

Update:

```
policies/vap/allowlist/allowed-operators-configmap.yaml
```

After updating the configuration, regenerate the RHACM policies using PolicyGenerator and apply the updated governance content.

---

## Audit Policy

The optional audit policy monitors changes to operator guardrail resources.

* Runs using `validationActions: [Audit]`
* Does not block changes
* Records violations in Kubernetes API server audit logs
